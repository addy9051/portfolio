Business Context
The platform, the funnel, and why a single step mattered so much
BA

An online micro-loan platform underwrites small consumer loans by running an applicant's bank statement through an LLM that summarises income, outflows, and risk signals before an offer is made. Revenue comes from the margin on funded loans, so every applicant who starts but never finishes the AI analysis is lost acquisition spend with nothing to show for it.

The funnel is six steps: sign_up → bank_link → ai_analysis_initiated → ai_analysis_completed → loan_terms_offered → loan_accepted. Product flagged that roughly a third of users vanish between analysis initiated and completed — the single largest leak in the funnel, and the one nobody could explain.

Problem Statement
What was broken, and what it cost to leave it alone
BADA

The AI-analysis step lost ~33% of users, but the team had no idea why — was it a UX problem, a model-quality problem, or an infrastructure problem? Without a diagnosis, every proposed fix was a guess, and any "fix" risked speeding things up at the expense of credit quality.

Ambiguity
No one could attribute the drop to a cause, so prioritisation stalled — UX, ML, and infra each blamed the others.
The trap
Any speed fix could silently raise the loan-default rate — unacceptable in a regulated lending context. The fix needed a hard credit-quality guardrail.

Root Cause Analysis
Diagnosing the drop honestly — including the confounder that biases the naïve answer
DADS
🔍
Latency cohorts surface the signal
Bucketing AI-analysis sessions by LLM latency, completion craters monotonically: 86.4% (<3s) → 78.9% (3–5s) → 63.1% (5–8s) → 43.3% (>8s). The >8s cohort completes at roughly half the rate of the fast cohort. The drop-off is latency-driven, not random impatience.
⚠
…but the naïve estimate is confounded
A naïve "latency → completion" regression is biased: requested_amount is a common cause. Bigger loans → richer bank statements → more tokens → higher latency, and bigger asks → more committed applicants → higher completion. So high-latency requests come from more-committed users, masking latency's true harm.
🎯
Back-door adjustment recovers the true effect
Adjusting for requested_amount (the back-door criterion) moves the latency coefficient from −0.264 (naïve) → −0.298 (+12.8% stronger) — confirming the naïve number understates the damage. The synthetic data was deliberately built with this confounder so the adjustment does real work rather than recovering a baked-in number.

The fix — a latency-fallback router. The gateway calls the primary model (llama3.2) with a hard 4-second budget; if it blows the budget it falls back to a fast model (llama3.2:1b). Every call is traced in Langfuse with model, tokens, latency, and a notional cost, grouped by a per-analysis session_id so the timed-out primary and the fallback land in one trace.

Key Findings & Results
The fix works — and it's validated rigorously, not just asserted
DADS
+9.2pp
The 2×2 factorial fix
Progress bar +3.3pp, fallback router +8.5pp, both +9.2pp completion — with a significant sub-additive interaction (the two fixes share some lift). The interaction is exactly why a 2×2 beats two separate A/Bs.
4.8×
The router cuts cost AND lifts completion
LLM cost per converted user: primary $0.036 vs fast $0.0076. The fast fallback is ~4.8× cheaper per conversion and faster (so it abandons less) — the core LLM-ops result.
+0.64pp
Default-rate guardrail holds
One-sided, fixed-horizon non-inferiority test: the 95% upper bound on the default-rate increase is +0.64pp, below the pre-registered 1.0pp margin → non-inferior. Speed didn't silently degrade credit quality.
+1.25pp
Fair-lending guardrail holds
A second non-inferiority test checks the router doesn't widen the prime−subprime completion gap. 95% UB +1.25pp < 2.0pp margin → non-inferior. Falsifiable: an injected disparity exists, so this can fail.

The ship decision

The recommendation gates on three conditions at once: completion improves and the default-rate guardrail holds and the fairness guardrail holds. All three pass → SHIP. Supporting rigor — Sample-Ratio-Mismatch χ² (fires on an injected bad split), an A/A simulation (false-positive rate ≈ 5%, uniform p-values), Benjamini-Hochberg multiple-testing correction, and a CUSUM monitor that catches an injected latency regression in 1 day — all run in the analysis layer and surface on the dashboard.

Reflections & What I'd Do Differently
The honest section most portfolios skip
DS
✓ What worked well
Injecting a real confounder so the back-door adjustment moves the estimate (−0.264 → −0.298) — the causal section demonstrates method value rather than recovering a number I planted.
Making the fairness guardrail falsifiable: an audit caught an earlier version that grouped by the wrong flag and was null-by-construction; rebuilding it as a real non-inferiority test on an injected disparity made it defensible.
Keeping the guardrails fixed-horizon while the primary metric can run sequential — and writing down why in an ADR — reads like production experimentation, not a notebook.
↺ What I'd do differently
Add a sequential mSPRT on the primary completion metric to show always-valid early stopping alongside the fixed-horizon guardrails.
Local Ollama is CPU-bound and slow, so the live gateway proves the mechanism on a few cases while scale analysis uses synthetic traces — a real deployment would fine-tune a small model or buy provisioned throughput instead.
Wire the fairness audit to true protected-class attributes (with consented data + a maturation window on defaults) rather than a credit-score proxy.
