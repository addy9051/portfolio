Business Context
What the business does, what data it produces, and why a lakehouse was the right foundation
DE

Atlon is a large manufacturer of sports equipment (cricket bats, badminton rackets, volleyballs, etc.) operating across several countries. For years their analytics ran on a mature Databricks platform — Bronze→Silver→Gold Delta layers feeding BI dashboards. Then Atlon acquired SportsBar Nutrition, a fast-growing startup in energy bars and athletic nutrition. Atlon's leadership chose to retain SportsBar's culture, teams, and processes — a respectful but risky decision.

The consequences became clear quickly. SportsBar's data was everywhere: spreadsheets, cloud drives, and hastily built APIs. Data formats didn't match. Reporting cycles didn't align. Some months of SportsBar's order data were simply missing — lost during their hyper-growth phase. As Atlon prepared to unify supply chain forecasting and inventory planning across both companies, the data chaos became a bottleneck. Metrics were inconsistent. Sales numbers conflicted. Nothing tied together.

"Give us a reliable data layer for both companies — something we can use until we fix this mess." — Bruce, COO, Atlon

Bruce, the COO, commissioned the data engineering team — led by Tony Sharma and executed by data engineer Peter — to solve the problem. Three success criteria were defined: unified and reliable analytics for both companies in a single dashboard; a low learning curve so newly hired analysts could pick it up quickly; and a scalable long-term architecture that could survive if the migration took two or three years instead of one.

Parent company (Atlon)
Established B2B sports equipment manufacturer. Gold layer in Databricks already existed — populated from historical CSVs (Jan 2024 – Nov 2025). Products: cricket, badminton, volleyball, etc. across a star schema. 93K+ transaction records.
Child company (SportsBar)
18 SKUs across 6 categories (Energy Bars, Protein Bars, Granola & Cereals, Recovery Dairy, Healthy Snacks, Electrolyte Mix). 35 B2B retail accounts across Bengaluru, Hyderabad, and New Delhi. Data in messy CSVs on S3.
Order data — SportsBar
51,810 rows (full load, Jul–Nov 2025) + 8,834 rows (incremental, Dec 2025). Daily grain in source → aggregated to monthly grain in Gold for consistency with Atlon's existing schema.
Data destination
Databricks Unity Catalog — fmcg catalog, 3 schemas: bronze, silver, gold. SportsBar data flows through all three layers; Atlon's data lives directly in Gold. All tables are Delta with Change Data Feed.

Problem Statement
Why querying raw CSVs directly is not a data strategy
DE

SportsBar's data was everywhere — spreadsheets, cloud drives, CSV dumps — with no schema contract. Four date formats in the same column, city names spelled nine different ways, a product name misspelled consistently across every protein record, negative prices, and non-numeric IDs. Meanwhile, Atlon's leadership needed a single dashboard showing unified metrics across both companies by end of November 2025.

No schema enforcement
S3 CSVs land with no validation. Date column contained yyyy/MM/dd, dd/MM/yyyy, yyyy-MM-dd, and dd-MM-yyyy — all in the same column across different batches.
No separation of raw vs clean
Without a Bronze layer preserving raw state and a Silver layer enforcing quality, there is no audit trail and no safe surface for analysts to query — every ad hoc query hits raw, unvalidated data.
No incremental strategy
Without an incremental load pattern, adding new data means re-processing the entire dataset. For growing order history this becomes prohibitively slow and expensive.
No surrogate keys
Raw IDs are not guaranteed numeric. customer_id and product_id both had non-numeric values in source. No consistent join key existed across tables.

Root Cause Analysis
Where the data quality issues actually came from
DE
📤
ERP exports have no format contract
Different export configurations or ERP upgrades over time produced inconsistent date formats. The same date column carries yyyy/MM/dd, dd/MM/yyyy, yyyy-MM-dd, and dd-MM-yyyy depending on when the batch was exported. There is no enforced output schema at the source.
✏
Free-text fields with no validation
City and product name fields are free-text in the source system. Human data entry produced 9 variants of 3 city names (Bengalore, Bengaluruu, Hyderbad, Hyderabadd, NewDelhi, NewDheli, NewDelhee, etc.) and a persistent "Protien" misspelling across all protein product records. These go undetected without a cleaning layer.
🎯
No data platform layer between source and consumers
The fix required a Medallion Architecture: Bronze preserves the raw source state exactly as received (with file metadata), Silver enforces schema, applies quality rules, and produces a validated dataset, and Gold applies business aggregations (daily → monthly grain) ready for reporting. Without this separation, every consumer builds their own cleaning logic — inconsistently.

Two key engineering decisions worth explaining

SHA2-256 surrogate keys for products
Raw product_id values are unreliable (non-numeric values fallback to "999999"). Instead of using the raw ID as a join key, a deterministic SHA2-256 hash of the product name is used as product_code. This is stable across data sources, survives ID changes in the ERP, and is collision-resistant for a 18-product catalogue.
Incremental load re-aggregates affected months
The incremental pattern doesn't just append new rows. It identifies which calendar months are touched by the new batch, extracts those months from the parent fact table, re-aggregates them including the new data, then merges back. This ensures monthly aggregates stay correct even if the new batch corrects earlier records.

Orchestration — Databricks Workflow

A 4-task Databricks Job was set up to run the incremental pipeline automatically. Tasks run in dependency order: Customers → Products → Pricing → Fact Orders (incremental). The job is scheduled at 11 PM daily (after business close) so each day's new order file is picked up from S3 landing, processed through Silver, and merged into the consolidated Gold tables overnight. Email alerts were configured for failure and success. Widgets on each notebook allow environment variables (catalog name, bucket path) to be passed at runtime without touching code.

Job structure (4 tasks)
1. Customer data processing → 2. Products data processing → 3. Gross price processing → 4. Fact orders (incremental load). Sequential dependency; task 2 depends on 1, etc.
Daily file pattern
New order CSV lands in S3 landing/ each night. Pipeline processes it, appends to Bronze, cleans into Silver staging, merges to Gold, then moves file to processed/ — preventing reprocessing on the next run.

Analytics Layer — Unified View, Genie & Dashboard

Above the Gold tables, a denormalized view (gold.vw_order_analysis) was created by joining all five Gold tables — fact orders with date, product, customer, and price dimensions. This single flat table exposes all columns analysts need (quarter, channel, category, total revenue = sold_quantity × gross_price INR, etc.) without requiring joins at query time.

Databricks Genie
Genie was configured against the unified view. Natural language queries such as "top 5 customers by sold quantity" or "total revenue by quarter" are converted to SQL automatically. Results render with auto-generated charts — no manual SQL needed for ad hoc analysis.
Databricks BI Dashboard
A dashboard was built with: KPI counters (total revenue, total quantity), interactive filters (year, quarter, month, channel, category), a top-10 products bar chart by revenue with tooltip for sold quantity, and a revenue share by channel pie chart. Filters refresh all visuals in place.

Reflections & What I'd Do Next
Honest assessment of what's solid and what the pipeline needs to reach production
DE
✓ What works well
The incremental load design is the most technically sound part of the project. Re-aggregating only affected months — rather than appending raw rows — means the monthly fact table stays correct even when an incremental batch corrects earlier records. This is the right approach for a monthly-grain analytical table.
Using SHA2-256 as the product surrogate key rather than the raw ERP ID was the right call. It's deterministic (same input always produces the same key), collision-resistant at this scale, and survives ERP ID changes without breaking joins downstream.
Bronze preserving the raw state exactly as received — including file metadata — means there is always an audit trail to diagnose quality issues. If a future Silver rule produces unexpected output, you can re-derive it from Bronze without going back to S3.
↺ What I'd strengthen
The data quality fixes are embedded in notebook cells with no automated validation. I'd add dbt tests (not-null, accepted values, unique, referential integrity) so that every Silver rule has a corresponding test that fails loudly at run time — rather than producing silent bad data that only surfaces in the dashboard.
The Databricks Workflow runs dimension tasks sequentially. Customers, Products, and Pricing are independent and could run in parallel — cutting the daily processing window roughly in half. This is a straightforward DAG change that would matter as data volumes grow.
Atlon's Gold tables were seeded from CSV files manually — there's no Bronze or Silver layer for the parent company. A production system would have Atlon's full pipeline too, feeding into the same consolidated Gold schema, rather than treating the parent data as a static starting point.
